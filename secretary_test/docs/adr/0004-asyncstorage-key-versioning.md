# ADR-0004: AsyncStorage 키 버전 패턴 (`{type}_v1_${userId}`)

## 컨텍스트

AsyncStorage에 스키마 변경이 생기면 기존 데이터를 마이그레이션해야 한다. SQL과 달리 `ALTER TABLE`이 없으므로 키 자체에 버전을 포함하는 방식이 필요하다. 또한 다중 계정(6개 테스트 계정)이 같은 기기에서 데이터를 독립적으로 유지해야 한다.

## 결정

모든 사용자별 데이터 키는 `{type}_v{n}_${userId}` 형식을 따른다.

```
schedules_v1_${userId}
clients_v1_${userId}
messages_v3_${userId}   ← 스키마 변경으로 v3까지 진행된 사례
```

글로벌 설정(사용자 무관)은 userId 없이 단순 키를 사용한다: `current_user_v1`, `claude_api_key`, `ai_provider`

## 이유

- 스키마가 바뀌면 새 버전 키를 사용하면 된다. 구 버전 데이터는 자동으로 무시된다 (별도 마이그레이션 코드 불필요).
- userId 접미사로 계정 간 데이터 충돌이 원천 차단된다.
- `messages_v3`처럼 버전 히스토리가 키 이름에 남아 변경 이력을 추적할 수 있다.

## 결과

- 긍정: 마이그레이션 코드 없이 스키마 변경 가능, 계정 데이터 격리 보장
- 부정: 구 버전 키 데이터가 스토리지에 남아 누적됨 (앱 용량 증가 가능)
- 제약: **이 패턴을 위반하는 키 사용은 데이터 손실로 이어진다.** 신규 키 추가 시 반드시 CLAUDE.md "AsyncStorage 키 목록"에 등록한다.
