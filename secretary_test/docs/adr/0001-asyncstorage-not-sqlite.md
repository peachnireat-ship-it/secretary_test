# ADR-0001: SQLite 대신 AsyncStorage 사용

## 컨텍스트

secretary_test는 서버가 없는 로컬 전용 앱이다. 저장 계층을 선택할 때 Expo에서 지원하는 두 가지 옵션이 있었다:
- `@react-native-async-storage/async-storage` — 키-값 JSON 저장소
- `expo-sqlite` — 관계형 SQL 데이터베이스

## 결정

AsyncStorage를 사용한다. 모든 데이터는 JSON 배열로 직렬화하여 사용자 ID별 키에 저장한다.

## 이유

- 이 앱의 데이터 접근 패턴은 "전체 목록 로드 후 메모리에서 필터링"이다. 복잡한 JOIN이나 집계 쿼리가 없다.
- Expo 관리 워크플로우(ADR-0003)에서 추가 네이티브 설정 없이 동작한다.
- 데이터 볼륨이 수천 건 수준으로 작아 전체 로드 후 JS 필터링 성능이 충분하다.

## 결과

- 긍정: 의존성 최소, Expo SDK 기본 포함, 설정 불필요
- 부정: 복잡한 쿼리 불가, 마이그레이션 수동 처리 필요 (키 버전 패턴으로 완화 — ADR-0004)
- 수용: 데이터 볼륨이 커지면 SQLite 전환을 재검토한다
