## 레이어별 설계 원칙

데이터 파이프라인 스키마는 3레이어로 분리한다. 레이어 간 혼합은 파이프라인 복잡도를 폭발적으로 증가시킨다.

| 레이어 | 목적 | 원칙 |
|--------|------|------|
| **Raw** | 소스 원본 보존 | 변환 없이 그대로. 컬럼명도 소스 그대로. 파티션만 추가. |
| **Cleansed** | 타입·형식 정제 | 타입 변환, 이상값 처리, 표준화. 비즈니스 로직 없음. |
| **Curated** | 비즈니스 서빙 | 집계, JOIN, 비즈니스 메트릭. 쿼리 최적화 형태. |

## 컬럼 타입 선택 기준

| 데이터 유형 | 권장 타입 | 주의사항 |
|-----------|---------|---------|
| 날짜 | TIMESTAMP (UTC) | DATETIME 금지 — 시간대 문제 발생 |
| 금액 | DECIMAL(18,4) | FLOAT 금지 — 부동소수점 오차 |
| ID/코드 | STRING | INT로 저장 시 선행 0 소실 위험 |
| Boolean | BOOLEAN | 0/1 정수 대신 명시적 타입 사용 |
| JSON 구조 | STRING + 별도 파싱 컬럼 | JSON 컬럼은 쿼리 성능 저하 |

## 메타데이터 컬럼 표준

모든 테이블에 필수 포함:
```sql
_ingested_at    TIMESTAMP  NOT NULL  -- 파이프라인 수집 시각
_source         STRING     NOT NULL  -- 소스 시스템 식별자
_pipeline_run_id STRING    NOT NULL  -- 실행 ID (재처리 추적용)
_is_deleted     BOOLEAN    DEFAULT FALSE  -- 소프트 삭제
```

## 파티셔닝 전략

파티션 키는 쿼리의 WHERE 절에서 가장 자주 사용되는 컬럼으로 선택한다.

| 데이터 특성 | 권장 파티션 키 |
|-----------|-------------|
| 시계열 데이터 | `date(created_at)` 일별 파티션 |
| 사용자별 분석 | `user_id % N` 해시 파티션 (N=100~1000) |
| 지역별 분석 | `region_code` 목록 파티션 |
| 혼합 | 날짜 + 보조 키 복합 파티션 |

파티션 당 크기 목표: 100MB~10GB. 이 범위를 벗어나면 파티션 전략 재검토.

## 스키마 진화 전략

스키마는 배포 후에도 변한다. 변경 유형별 영향도:

| 변경 유형 | ETL 영향 | 안전도 | 처리 방법 |
|---------|---------|-------|---------|
| 컬럼 추가 (nullable) | 없음 | 🟢 안전 | 즉시 적용 |
| 컬럼 추가 (not null) | 있음 | 🟡 주의 | 기본값 설정 후 적용 |
| 컬럼 타입 변경 | 있음 | 🔴 위험 | 신규 컬럼 추가 후 마이그레이션 |
| 컬럼 삭제 | 있음 | 🔴 위험 | ETL 수정 후 deprecated 표시 → 삭제 |
| 파티션 키 변경 | 있음 | 🔴 위험 | 새 테이블 생성 + 데이터 마이그레이션 |

## ETL과의 협업 포인트

- ETL이 요청하는 비정규화가 타당한 경우: 반정규화(Denormalization)를 Curated 레이어에만 허용
- 스키마 변경 통보 시: 변경 컬럼명, 이전 타입, 새 타입, 적용 시각을 SendMessage로 전달
- Raw 레이어 컬럼명은 소스 원본 그대로 유지 (ETL이 변환할 책임)
