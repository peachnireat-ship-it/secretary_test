## 검증 규칙 분류 체계

검증 규칙은 4가지 유형으로 분류하고, 각각 측정 방법과 임계값을 명시한다.

| 유형 | 설명 | 예시 |
|------|------|------|
| **완전성** (Completeness) | 필수 값 존재 여부 | `null_rate(order_id) = 0` |
| **정확성** (Accuracy) | 값이 허용 범위 내 | `0 ≤ amount ≤ 10,000,000` |
| **일관성** (Consistency) | 레코드 간 논리적 일관성 | `end_date ≥ start_date` |
| **적시성** (Timeliness) | 데이터 신선도 | `max(created_at) ≥ NOW() - 1h` |

## 레이어별 검증 시점

검증은 데이터를 받는 즉시 실행한다. 나중에 몰아서 검증하면 오류의 영향 범위가 커진다.

```
[소스] → Extract → [Raw 검증 ①] → Transform → [Cleansed 검증 ②] → Aggregate → [Curated 검증 ③]
```

**① Raw 레이어 검증 (수신 직후):**
- Not Null 검사 (PK, 필수 FK)
- 허용 타입 범위 (최솟값/최댓값)
- 중복 레코드 탐지 (PK 기준)

**② Cleansed 레이어 검증 (변환 후):**
- 비즈니스 규칙 (음수 금액, 미래 날짜 등)
- 참조 무결성 (외래키가 참조 테이블에 존재하는가)
- 변환 손실 여부 (변환 전후 레코드 수 비교)

**③ Curated 레이어 검증 (집계 후):**
- 집계 결과 일관성 (Raw 합계 = Curated 합계)
- 기대 레코드 수 (특정 기간 데이터 무결성)
- 전일 대비 이상 감지

## 임계값 설정 가이드

임계값은 과거 데이터 기반으로 설정한다. 과거 데이터가 없으면 초기값을 설정하고 2주 후 실제 분포 기반으로 재검토한다.

| 상황 | 임계값 설정 방법 |
|------|--------------|
| 과거 데이터 있음 | 평균 ± 3σ (99.7% 신뢰구간) |
| 과거 데이터 없음 | 비즈니스 상식 기반 초기값 + "⚠️ 2주 후 재검토" 태그 |
| SLA 요구사항 있음 | SLA에서 역산 (예: "99.9% 정확도" → null_rate < 0.001) |

## 위반 시 처리 정책

| 심각도 | 기준 | 처리 방법 |
|-------|------|---------|
| 🔴 Critical | PK null, 레코드 수 0, 집계 불일치 | 파이프라인 즉시 중단 + 온콜 알림 |
| 🟡 Warning | 이상값 비율 초과, 볼륨 이상 | Dead Letter Queue 격리 + 팀 알림 |
| 🟢 Info | 비표준 형식, minor 이상 | 로깅 + 대시보드 표시 |

**Dead Letter Queue 설계:**
격리된 레코드는 `_dlq_{테이블명}` 테이블에 저장:
- 원본 레코드 전체
- 위반 규칙 ID
- 격리 시각
- `_pipeline_run_id` (추적용)

## 도구 독립적 규칙 작성

규칙은 도구에 종속되지 않게 작성한다. 이후 Great Expectations, dbt tests, Soda 등 어느 도구로도 구현 가능하다:

```
규칙: order_id는 null 불가
측정: SELECT COUNT(*) FROM raw_orders WHERE order_id IS NULL
기대값: 0
위반 시: Critical, 파이프라인 중단
```
