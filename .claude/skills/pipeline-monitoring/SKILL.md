## 골든 시그널 (Four Golden Signals)

데이터 파이프라인 모니터링의 핵심 4가지 지표:

| 시그널 | 파이프라인 적용 | 측정 방법 |
|-------|-------------|---------|
| **Latency** (지연) | 파이프라인 완료 소요 시간 | `완료_at - 시작_at` |
| **Traffic** (처리량) | 단위 시간당 처리 레코드 수 | `COUNT(records) / elapsed_minutes` |
| **Errors** (오류율) | 실패 레코드 / 전체 레코드 | `failed_count / total_count` |
| **Saturation** (포화도) | 자원 사용률 (CPU, 메모리, 디스크) | 인프라 모니터링 연동 |

이 4가지를 기본으로 하고, 도메인별 비즈니스 지표를 추가한다.

## SLA 역산 방법

SLA를 선언하고, SLA에서 역산하여 각 단계의 완료 기대 시각을 정의한다.

```
SLA: D+1 09:00까지 전일 데이터 집계 완료
→ Curated 적재 완료: 08:45 (여유 15분)
→ Cleansed 변환 완료: 08:00 (45분 처리 예상)
→ Raw 수집 완료: 07:00 (1시간 처리 예상)
→ Extract 시작: 06:00 (1시간 추출 예상)
```

각 단계에 SLA 마감 시각을 명시하면, 초과 시 알림을 자동화할 수 있다.

## 알림 계층화 원칙

알림이 너무 많으면 온콜 담당자가 알림을 무시하기 시작한다 (알림 피로). 계층화로 노이즈를 줄인다.

| 계층 | 기준 | 알림 채널 | 예시 |
|------|------|---------|------|
| 🔴 P1 Critical | 즉각 대응 필요, SLA 위협 | PagerDuty/전화 | 파이프라인 완전 실패, 데이터 0건 |
| 🟡 P2 Warning | 업무 시간 내 대응 | Slack #alerts | 처리 지연 30분 초과, 볼륨 이상 |
| 🟢 P3 Info | 정기 확인 | 대시보드만 | 일별 처리 완료, 검증 통과율 |

**에스컬레이션 정책:**
P1 발생 → 온콜 5분 응답 없음 → 팀 리더 알림 → 15분 응답 없음 → 부서장 알림

## 볼륨 이상 탐지

조용한 실패(데이터 0건인데 파이프라인은 "성공") 탐지:

```
이상 탐지 규칙:
- 전일 동시간대 볼륨 대비 50% 미만 → Warning
- 전일 동시간대 볼륨 대비 10% 미만 → Critical
- 7일 이동평균 대비 3σ 이상 이탈 → Warning
```

요일 효과가 있는 데이터(주말 감소 등)는 "동시간대 N주 평균"으로 비교해야 한다.

## 대시보드 구성 가이드

대시보드는 목적별로 분리한다:

| 대시보드 | 대상 | 주요 패널 |
|---------|------|---------|
| 운영 현황 | 온콜 담당자 | 파이프라인 상태, 지연 현황, 활성 알림 |
| 데이터 품질 | 데이터 엔지니어 | 검증 통과율, DLQ 현황, 이상값 추이 |
| 비즈니스 SLA | 팀 리더 | SLA 달성률, 처리량 추이, 다음 처리 예정 |

**대시보드 패널 기본 구성:**
```
[파이프라인 현재 상태: 실행중/완료/실패]
[오늘 처리량 vs 전일 처리량]
[단계별 소요 시간 (Gantt 형태)]
[최근 N건 실패 목록 + 오류 메시지]
[검증 규칙 통과율 히트맵]
```

## 런북(Runbook) 작성 원칙

런북은 "모르는 사람도 따라할 수 있어야" 한다. 전제 지식을 가정하지 않는다.

**런북 필수 항목:**
1. 알림 조건 (어떤 상황에서 이 런북을 보는가)
2. 초기 진단 명령어 (실행하면 원인 파악 가능한 쿼리/명령)
3. 원인별 해결 절차 (if A → do X, if B → do Y)
4. 에스컬레이션 기준 (혼자 해결 불가 시 누구에게 연락하는가)
5. 복구 확인 방법 (파이프라인이 정상 복구됐음을 어떻게 확인하는가)
