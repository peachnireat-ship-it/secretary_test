## 종합 점수 계산 공식

```
종합 점수 = (아키텍처 × 0.25) + (보안 × 0.35) + (성능 × 0.25) + (스타일 × 0.15)
```

가중치 근거:
- **보안(0.35)**: 취약점은 즉각적 위험, 악용 가능성 있음
- **아키텍처(0.25)**: 구조 문제는 모든 기능에 영향
- **성능(0.25)**: 사용자 경험 직결
- **스타일(0.15)**: 유지보수성에 영향, 즉각적 위험 낮음

## 교차 분석 패턴

```
# 연관 패턴 탐지 방법
1. 각 감사 파일에서 언급된 파일 경로 목록 추출
2. 2개+ 감사에서 중복 언급된 파일 → 핫스팟
3. 관련 이슈들의 공통 원인 탐색

# 복합 문제 예시
아키텍처: "controllers/order.ts — 비즈니스 로직 혼재 (Major)"
보안: "controllers/order.ts — userId 검증 없음 (Critical)"
→ 교차 분석: "계층 분리 부재가 인가 로직 누락으로 이어짐 — 구조 수정 없이 보안 수정만 해도 재발 위험"
```

## 우선순위 정렬 기준

```
1순위: Critical 취약점 (보안) — 배포 전 반드시 수정
2순위: Critical (아키텍처/성능) — 이번 스프린트 내 수정
3순위: Major (전 영역) — 다음 스프린트
4순위: Minor — 기술 부채 백로그 등록
```

## Executive Summary 작성 원칙

3줄 이내, 비기술 독자도 이해 가능하게:

```
나쁜 예: "순환 복잡도 위반 3건, DRY 위반 7건, N+1 쿼리 2건 발견"
좋은 예: "보안 취약점 2건(Critical)이 발견되어 즉시 수정이 필요합니다. 
          성능 병목 3건은 트래픽 증가 시 서비스 장애로 이어질 수 있습니다.
          코드 구조는 전반적으로 양호하나 일부 파일 리팩터링이 권장됩니다."
```

## 리포트 구조

```markdown
# 종합 코드 리뷰 리포트

## Executive Summary
(3줄 이내 — 비기술 독자 대상)

## 영역별 점수 대시보드
(테이블 + 시각적 바 표현)

## 핫스팟 파일
(여러 영역에서 문제 발견된 파일 우선 목록)

## 교차 분석 — 복합 문제
(단일 감사로 보이지 않는 연관 패턴)

## 즉시 조치 (이번 주)
## 다음 스프린트
## 기술 부채 백로그

## 긍정 평가
## 수정 로드맵
```
